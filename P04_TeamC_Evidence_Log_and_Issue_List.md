# P04 — Evidence Log and Issue List

## 1. Evidence Log
บันทึกหลักฐานที่ได้จากการรวบรวมข้อมูล (จากการสัมภาษณ์) ในสถานการณ์ "คำขอเร่งด่วนขัดกับกฎ"

| Evidence ID | Source (แหล่งที่มา) | Description (รายละเอียดข้อเท็จจริงที่พบ) |
|---|---|---|
| EV-C01 | คุณแอน (เจ้าหน้าที่) | ได้รับสายโทรศัพท์จากนักศึกษาขอใช้ห้องนอกเวลาราชการอย่างเร่งด่วน โดยไม่มีแบบฟอร์มหรือช่องทางออนไลน์รองรับคำขอลักษณะนี้ ทำให้ต้องตัดสินใจเฉพาะหน้า |
| EV-C02 | คุณแอน (เจ้าหน้าที่) | ระบุว่าไม่ต้องการเป็นผู้ปฏิเสธคำขอเร่งด่วนด้วยตนเองผ่าน LINE เพราะมักถูกผู้ใช้ตื๊อหรือต่อว่า ต้องการให้ระบบเป็นตัวกรองกฎเบื้องต้น |
| EV-C03 | คุณแนน (เจ้าหน้าที่/ผู้ดูแลนโยบาย) | ห้องปฏิบัติการ (Lab) นอกเวลาราชการจะไม่อนุมัติเด็ดขาด เว้นแต่มีชื่ออาจารย์ที่ปรึกษาควบคุมดูแลกิจกรรมและเซ็นรับรองความปลอดภัย |
| EV-C04 | นักศึกษา (ต้น) | หากไม่มีการแจ้งผลการอนุมัติห้องด่วนภายใน 2 ชั่วโมงก่อนเวลากิจกรรม ทีมงานจะไม่สามารถจัดเตรียมสถานที่หรือแจ้งผู้เข้าร่วมงานได้ ทำให้งานล่ม |
| EV-C05 | อาจารย์ศุภชัย (ผู้อนุมัติ) | คำขอเร่งด่วนนอกเวลาราชการต้องมีการบันทึกเหตุผลความจำเป็นอย่างละเอียด เพื่อเก็บเป็นหลักฐานย้อนหลัง (Audit Trail) หากเกิดกรณีทรัพย์สินเสียหายหรืออุบัติเหตุ |

## 2. Issue List
รายการปัญหาที่พบจาก Evidence Log นำมาจัดหมวดหมู่เพื่อใช้หาความต้องการของระบบ (Requirements) ต่อไป

| Issue ID | Issue Description (รายละเอียดปัญหา) | Impact (ผลกระทบ) |
|---|---|---|
| ISS-C01 | **No Formal Exception Channel:** ไม่มีระบบหรือแบบฟอร์มรองรับการยื่นขอข้อยกเว้นนอกเวลาราชการอย่างเป็นทางการ ผู้ใช้ต้องโทรตามหรือส่ง LINE เร่งด่วน | เจ้าหน้าที่ต้องตัดสินใจหน้างานโดยไม่มีเกณฑ์รองรับที่ชัดเจน เสี่ยงต่อความผิดพลาดและความไม่เป็นธรรมในการพิจารณา |
| ISS-C02 | **Undefined "Urgent" & Missing SLA:** ยังไม่มีนิยามหรือเกณฑ์วัดคำว่า "เร่งด่วน" และไม่มีกำหนดเวลาตอบกลับ (SLA) ที่ชัดเจน | ผู้ขอใช้งานไม่ทราบว่าต้องรอผลนานเท่าใด เสี่ยงที่กิจกรรมจะล่มหากแจ้งผลไม่ทัน |
| ISS-C03 | **Decision Burden on Staff:** เจ้าหน้าที่ต้องแบกรับความเสี่ยงในการตัดสินใจหรือปฏิเสธคำขอด้วยตนเองโดยไม่มีระบบช่วยกรองกฎเบื้องต้น | เจ้าหน้าที่เผชิญแรงกดดันและถูกต่อว่าจากผู้ใช้งานโดยตรง เกิดความล่าช้าเนื่องจากต้องเช็กหลายฝ่ายเอง |
| ISS-C04 | **Lack of Audit Trail:** ไม่มีการบันทึกเหตุผลความจำเป็นและผลการอนุมัติของคำขอเร่งด่วนอย่างเป็นระบบ | เสี่ยงต่อการตรวจสอบย้อนหลังหากเกิดอุบัติเหตุหรือทรัพย์สินเสียหาย ไม่มีหลักฐานยืนยันกระบวนการตัดสินใจ |

## 3. Fact vs Interpretation
การแยกแยะระหว่าง "ข้อเท็จจริง (Fact)" ที่เกิดขึ้น กับ "การตีความ/ความเข้าใจ (Interpretation)" ของผู้เกี่ยวข้อง เพื่อลดอคติในการออกแบบระบบ

| เหตุการณ์ / สิ่งที่พบ | Fact (ข้อเท็จจริง) | Interpretation (การตีความของ Stakeholder) |
|---|---|---|
| **การรับคำขอของเจ้าหน้าที่** | คุณแอนรับสายโทรศัพท์ขอใช้ห้องนอกเวลาโดยไม่มีแบบฟอร์มหรือระบบรองรับคำขอประเภทนี้ | **เจ้าหน้าที่ตีความว่า:** ตนเองไม่มีอำนาจตัดสินใจอนุมัติเอง จึงเลี่ยงที่จะตอบตกลงทันทีเพื่อป้องกันความเสี่ยงต่อตนเอง |
| **นโยบายข้อยกเว้น** | ป้ายประกาศ/นโยบายคณะระบุเพียงว่า "ยกเว้นตามเงื่อนไขที่กำหนด" โดยไม่มีรายละเอียดเงื่อนไขแนบมาด้วย | **นักศึกษาตีความว่า:** กิจกรรมของตนน่าจะเข้าเกณฑ์ยกเว้นได้ ในขณะที่ **เจ้าหน้าที่ตีความว่า** ต้องส่งให้ผู้อนุมัติพิจารณาเป็นรายกรณีเท่านั้น |
| **ห้องปฏิบัติการ (Lab) นอกเวลา** | มีการระบุชื่ออาจารย์ที่ปรึกษาในคำขอ แต่ยังไม่มีการเซ็นรับรองความปลอดภัยจริง | **นักศึกษาตีความว่า:** การระบุชื่ออาจารย์เพียงพอแล้วสำหรับการอนุมัติ ขณะที่ **ผู้อนุมัติตีความว่า** เงื่อนไขยังไม่ครบถ้วนจึงไม่สามารถอนุมัติได้ |
| **ระยะเวลาการแจ้งผลอนุมัติ** | ผู้อนุมัติยังไม่ได้ตอบกลับภายในกรอบเวลาที่นักศึกษากำหนดไว้ (2 ชั่วโมงก่อนกิจกรรม) | **นักศึกษาตีความว่า:** ระบบละเลยความเร่งด่วนของคำขอ ขณะที่ **ผู้อนุมัติตีความว่า** กำลังตรวจสอบเอกสารและความเสี่ยงตามขั้นตอนปกติ |
